## Context
当前线上部署在“服务器 pull 不稳定 + 本地/服务器架构差异”的情况下，容易退化为离线传镜像。离线传镜像对小改动（仅模板/标题）成本过高。

## Goals / Non-Goals
- Goals:
  - 给日常小改动提供 A 热修补路径，显著缩短上线时间。
  - 给标准发布提供 C CI+镜像仓库路径，降低服务器依赖外网的风险。
  - 固化流程：对小改动必须先询问并由你选择 A 或 C。
- Non-Goals:
  - 不在本 change 中引入复杂蓝绿/灰度；先实现可用的快路径与标准发布路径。

## Decisions
- Decision: 将部署方式显式建模为两种模式（A/C），并在每次操作前做选择确认。
- Decision: A 模式只允许“静态资源/模板”的热更新，不允许改变依赖/后端逻辑。
- Decision: C 模式要求 CI 构建 `linux/amd64` 镜像，并推送到服务器可访问仓库；服务器端部署以 pull 为主，离线为 fallback。

## Risks / Trade-offs
- A 模式风险：热修补可能造成“线上与仓库不一致” → 约束：A 模式必须同时提交代码，且下一次 C 发布会覆盖热修补。
- C 模式成本：需要维护 CI/仓库凭据与网络可达性 → 但换来稳定快速的标准发布。

## Migration Plan
- 先落地 A 模式（最能立即提速）
- 再落地 C 模式（长期标准化）
